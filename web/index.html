Aegis

A firewall for AI agents that pay in USDC on Ethereum.

AI agents are starting to move real money on their own. The risk is that an agent sends funds straight into a honeypot contract, or approves a malicious spender that quietly drains the wallet later. Aegis checks who is about to receive the money and returns one verdict before the transaction is ever signed - and then enforces the same policy in a smart contract on-chain, so a compromised agent can't route around it.

PAY  safe to send BLOCK  dangerous, stop it ASK_HUMAN  not sure, escalate

What it checks

Three quick checks on the recipient, plus an LLM second opinion when one of them trips.

Bytecode REAL

Reads the recipient's on-chain code with eth_getCode and walks the EVM opcodes, flagging traps like SELFDESTRUCT, DELEGATECALL, CALLCODE, CREATE2, tx.origin and EIP-1167 proxies.

Reputation MOCK

Allow and deny address lists for now. Roadmap: a live reputation oracle built on on-chain settlement history.

Behavior MOCK

A large amount going to a first-time recipient gets escalated to a human. Roadmap: a proper spend-velocity model.

LLM reasoning REAL

When a check trips or isn't sure, Aegis asks an LLM to read the raw bytecode and either confirm a real trap or clear a false alarm. It runs locally through the Claude CLI, so there is no API key.

How it decides

denylisted            -> BLOCK
dangerous bytecode    -> LLM reads it -> BLOCK or ASK_HUMAN
unvetted contract     -> REVIEW
big unknown payment   -> ASK_HUMAN
all clear             -> PAY

If the LLM step ever fails, the engine falls back to the deterministic verdict, so the firewall never breaks open.

Enforced on-chain

The verdict isn't just advice. Agents pay through the AegisGuard contract, which re-checks the policy in Solidity before any USDC moves.

  • The funds live in a treasury that approves the guard and nothing else, so a leaked agent key can't move money outside the policy.
  • assess() and the transfer run in one call, so there's no gap for the recipient to swap its code between the check and the payment.
  • The three lanes map onto real contract state: a denylist / allowlist, a codehash kill-switch, and a rolling daily spend cap.
  • The two layers are wired together: when the analyzer is sure a recipient is malicious it writes that verdict into the guard - blocking the bytecode by codehash - so the contract enforces it even for an agent that never runs the off-chain check. The reverse, trusting a new contract, is deliberately gated behind an owner confirmation.

Recipient is a plain wallet with no contract code to run. Off-chain and on-chain both say PAY, so 10 USDC settled through the guard for real.

node src/agent.js guard bad
BLOCK

The recipient is a honeypot: a delegatecall proxy with an owner selfdestruct and tx.origin auth (the Parity-freeze shape). The scan flags it HIGH and the LLM confirms the rug. On-chain the guard starts at REVIEW (an unvetted contract) - so Aegis records the honeypot's codehash into the guard, and its own verdict flips REVIEW → BLOCK in the same run. Nothing was sent, and the guard now blocks that bytecode on its own.

ASK_HUMAN

Approving a plain wallet. A real spender is almost always a contract like a router or escrow, so an allowance to an EOA is a phishing red flag. Aegis pauses for a human instead of auto-approving.

The drainer guard. Agents usually lose funds by approving a bad spender, not just by sending. Aegis reads the spender's code and blocks the allowance.

Why Ethereum-native

  • USDC. Payments are real on-chain ERC-20 transfers, not off-chain IOUs.
  • Bytecode analysis. The core check reads the actual EVM instructions the recipient would run.
  • Enforced in Solidity. The policy lives in a verified contract, not just in the client, so it can't be skipped.
  • Approval-aware. It guards the approve() path, where most agent drains actually happen.
